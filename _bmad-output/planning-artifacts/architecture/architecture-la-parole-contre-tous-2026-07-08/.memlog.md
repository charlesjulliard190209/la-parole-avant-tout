- (event) Run démarré depuis le PRD final (prd-la-parole-contre-tous-2026-07-06). Charles signale être débutant et demande un accompagnement pas-à-pas pour les décisions -> mode Coaching retenu.
- (decision) Livrable : le spine technique seul suffit, pas de document annexe pour l'instant (Charles + Basile sont la seule audience).
- (decision) Wix abandonné entièrement : on construit un nouveau site unique (vitrine + chat + interface organisateurs) hors Wix, plutôt qu'un lien Wix→appli séparée ou un iframe. Choix de Charles (option 'Autre'), plus radical que les deux options proposées.
- (decision) Stack confirmée : Next.js (frontend + backend légers en un seul projet) + Supabase (comptes organisateurs, base de données Postgres) + Vercel (hébergement). Combinaison la plus documentée/tutoriellisée en 2026, pas de serveur à gérer soi-même.
- (constraint) Vercel Hobby (gratuit) = usage non-commercial personnel uniquement (vérifié web, juillet 2026) : pas de paiement/pub/dons monétisés sur le site tant qu'on reste sur ce plan gratuit — cohérent avec le projet (aucun organisateur n'en tire un revenu). Si ça change un jour, upgrade vers Pro nécessaire.
- (constraint) Supabase free tier (vérifié web, juillet 2026) : 500 Mo base de données, 1 Go stockage fichiers, 5 Go egress/mois, 50k utilisateurs actifs/mois, projet mis en pause après 7 jours d'inactivité totale (se réactive automatiquement à la prochaine requête, léger délai) — largement suffisant pour ce volume (2 organisateurs + élèves d'un seul lycée).
- (decision) Notification organisateurs (FR-7) : bot Telegram — le serveur POST vers l'API Telegram à chaque nouveau message ; sur Signal de danger (FR-10) et relance FR-15, même canal Telegram réutilisé (message répété/insistant plutôt que canal de secours séparé, plus simple pour deux débutants).
- (decision) Session anonyme + Code de récupération (FR-2, FR-17, FR-18) : mécanisme maison unique plutôt que la fonctionnalité Anonymous Sign-in de Supabase. Chaque conversation a un jeton secret aléatoire posé en cookie httpOnly (retour auto FR-2) ; le Code personnalisé (FR-17) est haché en base (jamais stocké en clair) et sert, une fois vérifié, à réémettre ce même jeton sur un nouvel appareil (FR-18). Un seul mécanisme à comprendre pour les deux cas, plus simple à déboguer pour une équipe débutante que de mélanger Supabase Auth anonyme + un système de code à côté.
- (decision) Liste de mots-clés Signal de danger (FR-9) : fichier de config versionné dans le code (pas de table+admin UI). Charles/Basile sont à la fois organisateurs et développeurs -> pas besoin d'interface dédiée pour l'instant. Modification = éditer le fichier + redéploiement (auto via push GitHub→Vercel).
- (version) Next.js 16.2.10 LTS (vérifié web, juillet 2026) ; @supabase/supabase-js 2.110.1 ; Tailwind CSS 4.3.2 ; Node.js 24 (LTS active) ; TypeScript (dernière stable, projet en TS par défaut avec Next.js).
- (decision) Authentification organisateurs (FR-4) : Supabase Auth email+mot de passe, exactement 2 comptes créés manuellement (dashboard Supabase), aucune page d'inscription publique dans le site.
- (decision) Convention mutations : Next.js Server Actions (App Router) pour toutes les écritures (envoi message, création/vérification Code, réponse organisateur) — pas de routes API séparées, sauf un appel serveur sortant vers l'API Telegram (pas un endpoint exposé). Idiome standard Next.js 2026, un seul patron à apprendre pour Charles/Basile.
- (decision) Anti-brute-force Code de récupération (FR-18) : compteur d'échecs + horodatage par tentative sur l'IP+code visé ; verrouillage temporaire (ex. 15 min) après 5 échecs consécutifs. Mécanisme simple en base, pas de service tiers.
- (decision) Environnements : un seul environnement de production (Vercel) + déploiements de prévisualisation automatiques par Vercel à chaque push (gratuits, inclus) — pas de vrai environnement de staging séparé à gérer, suffisant pour deux débutants sans clients externes en test.
