# Story 1.5: Récupération d'une conversation via Code

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a élève qui a créé un Code,
I want le saisir depuis n'importe quel appareil pour retrouver ma conversation,
so that je ne suis pas bloqué sur un seul appareil.

## Acceptance Criteria

1. Given j'ai un Code "Sauvegarder" valide (créé en Story 1.2), When je le saisis sur `/discussion-anonyme` depuis un appareil sans cookie `session_token` correspondant, Then ma conversation complète (historique élève + réponses organisateur) s'affiche, exactement comme le retour automatique par cookie (Story 1.4). [Source: epics.md#Story-1.5, prd.md#FR-18]
2. Given je saisis un Code invalide (inexistant, mal orthographié, ou appartenant à une conversation éphémère qui n'a jamais eu de Code), When je valide, Then un message d'erreur générique s'affiche — **identique** qu'un Code proche existe ou non (aucune indication de type "ce Code n'existe pas" vs "mauvais Code"). [Source: epics.md#Story-1.5, prd.md#FR-18, ARCHITECTURE-SPINE.md#Consistency-Conventions "Erreurs"]
3. Given j'ai échoué 5 fois consécutivement depuis la même IP en moins de 15 minutes, When je retente une saisie de Code (même correcte), Then la vérification est bloquée pour cette IP pendant 15 minutes — un message générique s'affiche, sans révéler si le Code lui-même était bon. [Source: epics.md#Story-1.5, ARCHITECTURE-SPINE.md#AD-9]
4. Given une récupération réussie (AC #1), When la conversation est retrouvée, Then un cookie `session_token` (httpOnly, ~12 mois) est posé sur ce nouvel appareil, permettant un retour automatique ultérieur (Story 1.4) sans re-saisir le Code à chaque fois. [Source: ARCHITECTURE-SPINE.md#AD-5]
5. Given une saisie de Code au format invalide (hors 6-20 caractères alphanumériques), When je valide, Then le même message d'erreur générique s'affiche sans qu'une tentative ne soit comptabilisée dans le compteur anti-brute-force (un format invalide ne peut de toute façon jamais correspondre à un Code réel — cohérent avec le comportement déjà établi de `choisirModeSauvegarder`, Story 1.2). [Source: ARCHITECTURE-SPINE.md#AD-9, app/discussion-anonyme/actions.ts (CODE_REGEX existant)]

### Review Findings

- [x] [Review][Defer] Verrou anti-brute-force contournable par requêtes concurrentes (TOCTOU) — `isRecoveryLocked` (lecture) et `recordRecoveryAttempt` (écriture) sont deux étapes non atomiques [lib/session.ts:154, 186]. **Décision de Charles (2026-07-09)** : risque accepté, même position que la race TOCTOU sur l'unicité du Code (Story 1.2) — fermer ce trou demanderait une transaction/verrou au niveau base pour un gain jugé faible à cette échelle.
- [x] [Review][Defer] Le verrouillage échoue "ouvert" (fail-open) sur toute erreur Supabase transitoire [lib/session.ts:154-179]. **Décision de Charles (2026-07-09)** : conservé tel quel, cohérent avec le reste du code (NFR-2, disponibilité), risque jugé acceptable.
- [x] [Review][Defer] Verrouillage partagé par IP brute — des élèves derrière la même IP (Wi-Fi/NAT du lycée) partagent le même compteur de tentatives [app/discussion-anonyme/actions.ts:210]. **Décision de Charles (2026-07-09)** : risque accepté, situation jugée rare en pratique, pas d'alternative simple sans identifiant supplémentaire par appareil.
- [x] [Review][Defer] Invalidation silencieuse du cookie d'un autre appareil à la récupération — `recupererConversationParCode` écrase `session_token_hash` sans afficher de message d'erreur sur l'appareil invalidé [app/discussion-anonyme/actions.ts:229-232]. **Décision de Charles (2026-07-09)** : laissé silencieux, cas jugé rare, pas d'action pour l'instant.
- [x] [Review][Patch] Le verrouillage se réinitialise si une réussite (même sur un Code différent) figure parmi les 5 dernières tentatives d'une IP — un attaquant possédant son propre Code valide peut l'intercaler pour ne jamais déclencher le verrou et deviner indéfiniment le Code d'une victime [lib/session.ts:167]. **Corrigé** : `isRecoveryLocked` compte désormais les échecs sur une fenêtre glissante de 15 minutes (requête `count` avec `eq("success", false)` + `gte("created_at", ...)`) plutôt que de regarder "les 5 dernières tentatives, toutes en échec" — une réussite intercalée ne réinitialise plus rien. Revérifié par script direct contre la vraie base (réussite intercalée parmi 5 échecs → toujours verrouillé).
- [x] [Review][Patch] `code_hash_attempted` est haché en bcrypt (coût CPU volontairement élevé) à chaque tentative alors qu'aucun code ne le relit jamais — un hachage rapide (sha256) suffit à respecter "jamais stocké en clair" sans le coût CPU inutile [lib/session.ts:195]. **Corrigé** : nouvelle fonction `hashForAudit` (sha256 via `crypto`) utilisée à la place de `hashSecret` (bcrypt) pour ce champ d'audit write-only. Revérifié par requête HTTP réelle : le hash stocké n'est plus au format bcrypt (`$2b$...`).
- [x] [Review][Patch] `recordRecoveryAttempt` est attendu (`await`) sur le chemin critique de la réponse alors que son résultat n'est jamais utilisé et ses erreurs sont avalées en interne — ajoute une latence évitable à chaque tentative [app/discussion-anonyme/actions.ts:220]. **Corrigé** : déferré via `after()` (`next/server`, disponible depuis Next 15, projet en 16.2.10) — s'exécute après l'envoi de la réponse. Revérifié par requête HTTP réelle : la ligne d'audit est bien écrite malgré le déferrement.
- [x] [Review][Patch] La séquence "générer un token → le hacher → le persister → poser le cookie" est dupliquée entre `choisirModeSauvegarder` et `recupererConversationParCode` sans fonction partagée — risque de dérive si l'une des deux est corrigée sans l'autre [app/discussion-anonyme/actions.ts:74-99,226-245]. **Corrigé** : nouvelle fonction `issueSessionToken()` (`lib/session.ts`) factorisant "générer + hacher", utilisée dans les deux Server Actions. Revérifié par `lint`/`build` + requête HTTP réelle de récupération de bout en bout (toujours fonctionnelle).
- [x] [Review][Defer] `recovery-form.tsx` duplique intégralement le balisage des deux autres formulaires du dossier (`mode-choice.tsx`, `message-form.tsx`) sans composant partagé — déjà différé, pré-existant : cohérent avec la préférence déjà documentée de ce dépôt pour la duplication à cette échelle plutôt qu'une abstraction prématurée (NFR-1), comme déjà tranché pour la triplication du scan bcrypt dans `lib/session.ts`.

## Dev Notes

- **Contrainte cryptographique centrale de cette story, à ne pas rater** : `ARCHITECTURE-SPINE.md#AD-5` dit "une fois vérifié, réémettre ce même `session_token` en cookie sur le nouvel appareil". Pris au pied de la lettre, c'est **techniquement impossible** — `session_token_hash` est un hachage bcrypt à sens unique (comme `recovery_code_hash`), le `session_token` d'origine n'est jamais récupérable depuis son hash. L'interprétation retenue dans `Task 3` (seule option réaliste avec les primitives déjà en place dans `lib/session.ts`) : générer un **nouveau** `session_token`, écraser `session_token_hash` en base pour cette Conversation, poser ce nouveau token en cookie sur l'appareil courant. Conséquence assumée : si l'élève avait un cookie valide sur un **autre** appareil pour cette même Conversation, ce cookie cesse de fonctionner après une récupération par Code ailleurs (un seul `session_token_hash` par Conversation, écrasé à chaque récupération). Aucune AC ne couvre ni n'interdit ce cas — voir question ouverte en fin de document.
- **Choix "fail-open" sur `isRecoveryLocked`** : en cas d'erreur Supabase transitoire, la fonction retourne `false` (non verrouillé) plutôt que de bloquer par prudence. C'est cohérent avec le reste du fichier (`findConversationBySessionToken`, Story 1.4, fait le même choix pour ne jamais empêcher un retour légitime — NFR-2), mais c'est aussi une fonction de **sécurité anti-brute-force** : un incident Supabase pendant une attaque désactiverait temporairement la protection plutôt que de bloquer les élèves légitimes. Décision déjà orientée par la philosophie du reste du code, mais à confirmer explicitement avec Charles (question ouverte en fin de document) plutôt qu'à trancher seul si un doute apparaît en cours d'implémentation.
- **Réconciliation de migration (Task 0), bloquant avant Task 1** : la migration de la Story 1.1 a été appliquée manuellement via le SQL Editor Supabase (pas par `supabase db push`, qui a échoué à cause d'un bug connu du pooler Supavisor) — voir `deferred-work.md`, section Story 1.1, qui pointe explicitement cette story pour la réconciliation. Ne pas ignorer cette étape : pousser une nouvelle migration sans réconcilier l'historique risque de recréer la même désynchronisation, ou pire, un échec silencieux qui laisserait `recovery_attempts` absente en production alors que le code qui en dépend (Task 2/3) est déployé.
- **Pattern de recherche O(n) par bcrypt, troisième occurrence** : `isCodeAvailable` (Story 1.2, unicité à la création) → `findConversationBySessionToken` (Story 1.4, retour par cookie) → `findConversationByRecoveryCode` (cette story, retour par Code). Même structure exacte à chaque fois (scan complet + `bcrypt.compare` par ligne, `try/catch` par ligne pour ne pas abandonner tout le scan sur une ligne corrompue). Ne pas factoriser ces trois fonctions en une seule generic — l'Architecture accepte la duplication à cette échelle (NFR-1, déjà tranché en Story 1.4 pour un cas similaire) ; introduire une abstraction ici serait un choix non demandé et non couvert par les AC.
- **`recovery_code_hash` n'existe que pour les Conversations "Sauvegarder"** (`is_ephemeral = false`) — la requête de `findConversationByRecoveryCode` n'a pas besoin de filtrer explicitement sur `is_ephemeral` : le filtre `recovery_code_hash is not null` suffit, exactement comme `isCodeAvailable` (Story 1.2) le fait déjà pour la vérification d'unicité à la création. Ne pas dupliquer un filtre `is_ephemeral = false` redondant.
- **Normalisation du Code (minuscules)** : `choisirModeSauvegarder` fait `.toLowerCase()` avant de hacher le Code à la création (Story 1.2) — un signalement de revue de la Story 1.3 note que rien n'avertit l'élève que son Code est insensible à la casse (`deferred-work.md`, entrée Story 1.3, non résolue, hors périmètre de cette story). `recupererConversationParCode` **doit** appliquer la même normalisation (`.toLowerCase()`) avant de comparer, sous peine de faire échouer silencieusement la récupération pour tout élève qui retape son Code avec une casse différente de celle saisie à la création.
- **IP cliente en Server Action** : pas de `request.ip` disponible côté Server Action Next.js App Router (ce n'est pas un Route Handler). Utiliser `headers()` de `next/headers` (déjà importée dans ce fichier via `cookies`) et lire `x-forwarded-for`, posé systématiquement par Vercel en production. En dev local, cet en-tête est généralement absent → repli sur la chaîne littérale `"unknown"`, ce qui revient à partager un seul "compteur" entre toutes les requêtes locales : sans conséquence en production, juste à garder en tête pendant les tests manuels (Task 6).
- **Le message "code invalide" (AC #2) et "format invalide" (AC #5) doivent être un seul et même texte constant**, pas deux chaînes qui se ressemblent — toute divergence, même mineure, réintroduirait le canal d'information que l'AD-9/la convention "Erreurs" de l'Architecture (`ARCHITECTURE-SPINE.md#Consistency-Conventions`) interdit explicitement pour le Code.
- **Aucune Server Action existante à modifier** : `envoyerMessage` et `choisirModeSauvegarder`/`choisirModeEphemere` restent inchangées. Cette story ajoute une fonction indépendante dans le même fichier `actions.ts` (cohérent avec la Capability Map de l'Architecture, qui assigne FR-18 à `app/discussion-anonyme/actions.ts` + `lib/session.ts`, aucun autre fichier).
